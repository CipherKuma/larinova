## Part 4 — Two-laptop sync strategy

This is the part you need to resolve before merging work from your other laptop.

### Current state

| Layer | Status |
|---|---|
| `landing/` | Git repo, 2 remotes, 4 modified + 1 new folder uncommitted |
| `mvp-web-app/` | Git repo, 2 remotes, ~7 modified + new migration `20260414000000_discovery_surveys.sql` uncommitted |
| Everything else | Not in git at all |

### The risk

If your other laptop has a different version of `docs/SALES_OUTREACH_PLAN.md`, or a different copy of the pitch PDFs, or new logos in `logo-gen/`, there is no automatic way to reconcile. You will either overwrite one laptop's work with the other or spend an afternoon diffing files manually. This has probably already happened.

### Recommended setup

**Three repos, one monorepo-lite folder:**

```
larinova/                       ← keep as the shared working folder, NOT a repo
├── CLAUDE.md                   ← move into larinova-ops (below)
├── landing/                    ← existing repo (keep)
├── mvp-web-app/                ← existing repo (keep)
└── larinova-ops/               ← NEW: one repo for everything else
    ├── docs/
    ├── sales/
    ├── logo-gen/
    ├── command-center/
    └── _archive/               ← or move _archive out entirely, it doesn't need two-laptop sync
```

The ops repo holds strategy, sales collateral, and brand assets. It is small (~13 MB including `_archive/`, or ~2 MB without). It will sync in seconds. It is private by default (these files include `indonesia_leads.xlsx` and unreleased pricing plans — do not make this public).

### Why this shape, not alternatives

- **"Make the root a git repo with submodules"** — Git submodules are painful in practice and add friction every time you switch branches in `landing/` or `mvp-web-app/`. Not worth it.
- **"Dropbox / iCloud for docs/sales"** — Tempting, but you lose history, PR-style review, and the ability to revert. And it conflicts badly with any file a CLI tool (or Claude) edits.
- **"One big monorepo"** — Overkill, and you'd have to untangle the existing `landing/` and `mvp-web-app/` histories. Not worth it.

```bash
# Compare laptop B's local docs/ with larinova-ops-remote/docs/ for each file

```

### Daily habit after setup

- Every time you sit down: `git pull` in all three repos first. Every time you stand up: `git push` in all three. Put this in a `sync.sh` if it helps.
- Never edit the same file on both laptops in the same session without a pull+push between them.
- When Claude (or any tool) edits files, commit the change before switching laptops. Uncommitted edits are the single biggest sync failure mode.

### Bonus: avoid the WIP-commit trap

Your current sub-repos have meaningful dirty work (new admin pages, calendar migration, LazyVideo component). Don't let those changes sit uncommitted on laptop A while laptop B has different dirty work on the same files. Even if a change is half-finished, a commit on a feature branch (`wip/admin-pages`) is safer than a dirty working tree.

If the question is *"are we on the right track for the Indonesia launch?"* — yes. The strategy is coherent:

1. **Offer is unified** (1 month free, no credit card, everywhere).
2. **Positioning is right** (native Bahasa Indonesia AI scribe, lead with language not price).

The risks are operational, not strategic:

- **Sync discipline** (Part 4) — fix before you touch either laptop again.
- **Pitch PDF bugs** (Part 3, #2-5) — fix before you hand any of these to another doctor or hospital.

One final honest thought: the Sigda Optima spec is the most valuable document you have. It proves you understand what Indonesian doctors actually do all day. Most scribe startups don't have that insight. Make sure whatever you build from here preserves it.
